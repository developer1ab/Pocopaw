# popopaw Project Roadmap 2026-05-19

## 0. Roadmap 边界

本文件只保留产品阶段、阶段目标和阶段顺序，不展开实现级细节。

白板中的 `包1 / 包2 / 包3 / 包4` 在本 roadmap 中统一改写为：

1. 阶段1：基础可用。
2. 阶段2：体验可接管。
3. 阶段3：记忆与复用增强。
4. 阶段4：联网与多模型扩展。

详细模块设计、实现约束和专题说明以下沉到 `design11` 系列文档为准。

## 1. 项目北极星

popopaw 的目标是成为一个 Android 端个人执行代理：用户以自然语言或其他输入形式表达目标，系统在尽量少打扰的前提下完成理解、准备、执行、验证、回答和长期复用。

整体推进顺序固定为：先可用，再好用，再聪明，最后联网增强。

## 2. 阶段式 Roadmap

### 2.1 阶段1：基础可用

目标：把 popopaw 做成一个稳定可用的基础产品。

本阶段重点：

1. 打通并稳定输入与执行前置准备。
2. 收口 Shizuku、无障碍和截图能力的基础 readiness。
3. 建立少 Clarify 的基础工作方式，让常见任务能直接回答、直接出方案或直接进入执行准备。

阶段结果：用户常见任务可以更稳定地进入回答、方案或执行主链，不再频繁卡在基础准备和无谓追问上。

### 2.2 阶段2：体验可接管

目标：把 popopaw 做成一个顺手、可观察、可接管的产品。

本阶段重点：

1. 增加语音等更自然的交互入口。
2. 强化风险分层、确认机制和权限边界。
3. 建立执行中的 Kiosk / 接管体验，让用户看得懂、接得住、退得出。

阶段结果：用户能理解系统当前在做什么、为什么停下、何时需要接管，系统也不会在高风险场景越权推进。

### 2.3 阶段3：记忆与复用增强

目标：把 popopaw 做成一个会积累、会复用、会逐步贴近用户的产品。

本阶段重点：

1. 建立更稳定的 memory、preference 和 process evidence 数据面。
2. 形成流程复用、默认路径排序和偏好驱动的候选增强。
3. 建立主动推荐与个性化能力，但保持 authority 边界不变。

阶段结果：系统能在不越权的前提下逐步积累偏好、复用流程、给出更贴近用户的默认建议。

### 2.4 阶段4：联网与多模型扩展

目标：把 popopaw 从本地执行代理扩展为带联网检索和多模型协同能力的综合代理。

本阶段重点：

1. 建立可解释的联网搜索回答链。
2. 明确搜索增强与执行 authority 的边界。
3. 引入多模型分工，但不形成平行控制面。

阶段结果：系统可以稳定地做联网检索、解释和回答，并在需要时继续回到受控的本地执行边界。

## 3. 横向原则

所有阶段都必须遵守以下原则：

1. passive 链条始终由用户最新显式语义驱动，状态信号不直接触发工作。
2. UI、Prompt、Task boundary、Runtime 的 authority 分层必须保持清晰。
3. 搜索、记忆、偏好、复用和主动性都属于 evidence 或策略增强层，不能替代 execution authority。
4. 高风险动作必须保留清晰的确认与安全边界。
5. 需要真机验证的能力必须覆盖 Samsung、OPPO，以及不同 readiness / 屏幕状态组合。

## 4. 专题池

roadmap 之外的具体专题，统一作为设计池管理，不再单独视为 roadmap 阶段。

| 专题 | 主要归属阶段 | 说明 |
| --- | --- | --- |
| 输入与执行准备 | 阶段1 | 输入通道、执行前置准备、少 Clarify 基础策略 |
| 体验与接管 | 阶段2 | 语音、安全边界、Kiosk 与人工接管 |
| 记忆与复用 | 阶段3 | memory、preference、process reuse、个性化与主动性 |
| 联网与模型扩展 | 阶段4 | 搜索回答链、多模型分工与联网能力扩展 |

## 5. 当前顺序

1. 先收口阶段1的范围和验收口径。
2. 阶段1稳定后，再进入阶段2的体验与接管增强。
3. 阶段3建立在稳定的 evidence plane 和流程复用基础之上。
4. 阶段4建立在稳定的搜索 contract 和多模型边界之上。
